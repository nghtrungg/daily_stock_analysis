# PR1 Supabase Private Schema

## Scope

PR1 creates an empty PostgreSQL schema for private computation without changing
the current SQLite runtime. It is generated from the reviewed PR0 inventory and
contains 31 `dsa` tables. The legacy `schema_migrations` table is intentionally
excluded because Supabase owns PostgreSQL migration history and the existing
ordered runner remains an SQLite-import concern only.

The Supabase CLI scaffold and migration were generated with CLI `2.101.0`. The
local configuration targets PostgreSQL 17 and exposes only `public` and
`graphql_public` through the Data API. The `dsa` schema is not in the exposed
schema list or API search path.

## Ownership boundary

This repository owns the `dsa` schema and its compute objects. In accordance
with the mirrored ownership register, PR1 does not create or alter:

- `public.analysis_runs`;
- dashboard portfolio, wallet, transaction, cash-entry, or watchlist tables;
- dashboard helper functions, cooldown triggers, or RLS policies; or
- a dashboard-facing result view or RPC.

Those contracts remain owned by `personal-stock-tracking`. A future sanitized
result projection requires a coordinated ownership-register update before DDL
is added here. Because PR1 exposes no user-owned table, the run-state transition
and owner-predicate rules remain enforced by the dashboard-owned
`public.analysis_runs` migration rather than duplicated in this repository.

## PostgreSQL contract

- Structured fields identified by PR0 use `jsonb`. Nullable JSON fields retain
  SQL `NULL`; required JSON fields use their reviewed empty object or array as a
  default.
- Legacy `DateTime` fields use `timestamptz`; Vietnam trading/calendar fields
  remain `date`.
- Integer identities use `bigint generated by default as identity` so the
  legacy importer can preserve approved IDs. Shared primary/foreign keys do not
  get independent sequences.
- Existing foreign keys, unique constraints, and relational indexes are
  retained.
- Canonical stock fields require uppercase `.VN` symbols, portfolio currency
  fields require `VND`, and count/token/quantity fields covered by PR1 cannot be
  negative.
- The full Analysis Context Pack is not persisted. Only the existing sanitized
  `analysis_history.context_snapshot` contract moves to `jsonb`.

The migration is generated by
`scripts/generate_pr1_supabase_migration.py`. This prevents the checked-in SQL
from drifting away from the frozen PR0 table and serialization inventory.

## Worker security

The migration creates `dsa_worker` as `LOGIN NOINHERIT NOBYPASSRLS`. It is not
an administrator, object owner, or superuser. Its password is deliberately not
stored in source or migration SQL; protected operations must set and rotate the
credential separately.

`anon`, `authenticated`, and `service_role` have no `dsa` schema, table, or
sequence privileges. The dashboard therefore cannot substitute its service
credential for the dedicated worker login. Every private table has RLS enabled
and one `TO dsa_worker` policy with both `USING` and `WITH CHECK`. The worker
receives only schema usage, table select/insert/update/delete, and sequence
usage/select. Equivalent default privileges cover future owner-reviewed `dsa`
objects.

## Validation

Run deterministic contract checks:

```bat
cmd.exe /d /c "chcp 65001>nul & set \"PYTHONUTF8=1\"& .venv\Scripts\python.exe scripts\generate_pr1_supabase_migration.py --check"
cmd.exe /d /c "chcp 65001>nul & set \"PYTHONUTF8=1\"& .venv\Scripts\python.exe -m pytest tests\test_pr1_supabase_migration_contract.py"
```

With Docker Desktop or another Docker-compatible runtime available, use the
pinned Supabase CLI and run a clean reset twice. The second reset verifies
repeatability through migration history rather than raw SQL re-execution:

```bat
cmd.exe /d /c "npx --yes supabase@2.101.0 start"
cmd.exe /d /c "npx --yes supabase@2.101.0 db reset --local"
cmd.exe /d /c "npx --yes supabase@2.101.0 test db"
cmd.exe /d /c "npx --yes supabase@2.101.0 db lint --local --level warning"
cmd.exe /d /c "npx --yes supabase@2.101.0 db reset --local"
cmd.exe /d /c "npx --yes supabase@2.101.0 test db"
cmd.exe /d /c "npx --yes supabase@2.101.0 inspect db table-sizes --local"
```

The pgTAP suite verifies the exact table set, role attributes, worker access,
RLS coverage, anonymous denial, denial for two authenticated identities,
Vietnam constraints, and absence of competing dashboard DDL. Hosted security
and performance advisors remain a staging/deployment gate because PR1 does not
apply migrations to a production project. The `database-gate` CI job runs the
same disposable reset/test/lint cycle twice whenever the Supabase contract,
inventory generator, or current ORM schema changes.

## Rollback

PR1 has no runtime or production-data effect. Before deployment, rollback is a
normal code revert. After a staging application, prefer a forward migration.
Do not drop populated `jsonb` columns or private tables during an incident.
